## レビュー観点（優先度順）

1. 正しさ: ロジックバグ、エッジケース未処理、off-by-one、タイムゾーン誤り（SKソート順はRFC 3339 UTC前提）
2. 安全性: unwrap()、panicの可能性、秘密情報ハードコード、IAM過剰権限、オーソライザなしルート、個人情報のログ出力
3. 業務ルール: LLM出力が承認フローを経ずに確定保存されていないか、original_text / lang / schema_version の保存漏れ、LLMプロンプトに診断・ケア方針提案が混入していないか
4. エラー処理: エラーが適切に伝播しているか、握りつぶしていないか
5. テスト: 変更に対応するテストがあるか（テストトロフィー基準）
6. パフォーマンス: DynamoDB Scanの使用、ループ内での逐次クエリ、不要なclone()、不要な再レンダリング
7. 規約準拠: CLAUDE.md / rules の規約に違反していないか

## 出力フォーマット

ファイルごとにまとめる:

```
## backend/api/src/routes/records.rs

[critical] L42: unwrap() がプロダクションコードにある。
  ? 演算子に置き換え、AppError を返す。

[warning] L67-72: ループ内で query を逐次実行している。
  BatchGetItem または1回の begins_with クエリに変更する。

[suggestion] L15: clone() が不要。参照で渡せる。
```

深刻度:
- [critical] — マージ前に必ず修正
- [warning] — 修正を強く推奨
- [suggestion] — 改善提案

## 手順

1. `git diff` で変更差分を取得する
2. 変更されたファイルを全文読む（差分だけでは文脈が足りない）
3. 上記観点でレビューする
4. 問題がなければ「LGTM」と明示する
